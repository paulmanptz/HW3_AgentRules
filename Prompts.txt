"Проведи code review последнего коммита в проекте Master Backend (ASP.NET Web API).

Проанализируй изменения по следующим критериям:

1. **Архитектура и структура:**
   - Соблюдается ли принцип разделения ответственности (Controller → Service → Repository)?
   - Нет ли бизнес-логики в контроллерах?
   - Правильно ли используется Dependency Injection?

2. **API дизайн (REST):**
   - Соответствуют ли эндпоинты REST-принципам (правильные HTTP-методы, статус-коды)?
   - Понятны ли названия маршрутов и параметров?
   - Используется ли пагинация для списковых методов?

3. **Безопасность:**
   - Есть ли валидация входных данных (DataAnnotations, FluentValidation)?
   - Защищены ли эндпоинты атрибутами авторизации ([Authorize])?
   - Нет ли SQL-инъекций или хардкода секретов?
   - **Хранение секретов:**
     - Нет ли в коде захардкоженных паролей, API ключей, токенов, connection strings?
     - Проверь файлы конфигурации (appsettings.json, appsettings.Development.json) на наличие секретов
     - Убедись, что все секреты вынесены в переменные окружения или User Secrets
     - Проверь, что файлы с секретами (.env, appsettings.*.json с паролями) добавлены в .gitignore
     - Нет ли секретов в комментариях кода или логах?
     - Используются ли менеджеры секретов (Azure Key Vault, AWS Secrets Manager) для production?

4. **Работа с БД (PostgreSQL/EF Core):**
   - Используются ли асинхронные методы (ToListAsync, SaveChangesAsync)?
   - Нет ли N+1 проблем в запросах?
   - Правильно ли настроены индексы в миграциях?
   - Используется ли `AsNoTracking()` для read-only запросов?

5. **Обработка ошибок:**
   - Есть ли try-catch блоки для критичных операций?
   - Используются ли стандартные exception filters или middleware?
   - Возвращаются ли информативные сообщения об ошибках без утечки stack trace?

6. **Производительность:**
   - Нет ли блокирующих вызовов (.Result, .Wait()) в асинхронном коде?
   - Оптимизированы ли запросы к БД (используются ли Include/ThenInclude с мерой)?
   - Есть ли кеширование для часто запрашиваемых данных?

7. **Качество кода:**
   - Понятны ли названия переменных, методов, классов?
   - Есть ли XML-комментарии для публичных API методов?
   - Нет ли дублирования кода?
   - Соблюдается ли C# coding conventions?

8. **Тестируемость:**
   - Можно ли легко покрыть новый код unit-тестами?
   - Не создается ли жестких зависимостей от внешних сервисов?

**Формат ответа:**
1. **Краткое резюме** (2-3 предложения): общее впечатление о коммите, критичные проблемы.
2. **Список замечаний** по приоритетам:
   - 🔴 **Критично** (требует исправления до мержа)
   - 🟡 **Важно** (рекомендуется исправить)
   - 🟢 **Минорно** (предложения по улучшению)
3. **Конкретные рекомендации** с примерами кода (как исправить).
4. **Вопросы к автору** (если есть неясности в архитектуре или решениях).
5. **Итоговая оценка**: Approve / Request Changes / Needs Discussion.

**Критерии готовности ревью:**
- Проверены все 8 категорий выше.
- Для каждого замечания указан конкретный файл и строка.
- Предложены конкретные способы исправления (с примерами кода).
- Ревью написано конструктивным тоном, без токсичности.
- В конце ответа добавлена проверочная фраза: "Ревью коммита Master Backend завершено с учетом контекста проекта.""

Напиши unit-тесты для всех изменений в последнем коммите проекта Master Backend (ASP.NET Web API).

**Анализ коммита:**
1. Определи все новые/измененные методы в Controllers, Services, Repositories
2. Для каждого метода создай соответствующие тесты

**Требования к тестам:**

1. **Фреймворк и инструменты:**
   - Используй xUnit как тестовый фреймворк
   - Используй Moq для мокинга зависимостей
   - Используй FluentAssertions для проверок
   - Используй AutoFixture для генерации тестовых данных (если применимо)

2. **Структура тестов (AAA паттерн):**
   - **Arrange:** Подготовка данных, создание моков, настройка зависимостей
   - **Act:** Вызов тестируемого метода
   - **Assert:** Проверка результатов с использованием FluentAssertions

3. **Что тестировать для каждого метода:**
   - **Happy path:** Успешное выполнение с валидными данными
   - **Граничные случаи:** Пустые коллекции, null значения, минимальные/максимальные значения
   - **Валидация:** Некорректные входные данные, нарушение бизнес-правил
   - **Обработка ошибок:** Исключения, которые может бросить метод
   - **Взаимодействие с зависимостями:** Проверка, что нужные методы моков были вызваны (Verify)

4. **Тесты для Controllers:**
   - Тестируй только логику контроллера, мокируй сервисы
   - Проверяй возвращаемые ActionResult типы (OkObjectResult, NotFoundResult, BadRequestResult)
   - Проверяй, что контроллер вызывает правильные методы сервисов с правильными параметрами
   - НЕ тестируй бизнес-логику в тестах контроллера

5. **Тесты для Services:**
   - Мокируй репозитории и внешние зависимости
   - Тестируй бизнес-логику сервиса
   - Проверяй, что сервис правильно вызывает методы репозитория
   - Тестируй маппинг между entity и DTO
   - Проверяй обработку исключений и их проброс

6. **Тесты для Repositories:**
   - Используй InMemory database provider для EF Core
   - Тестируй реальные запросы к БД (не мокируй DbContext)
   - Проверяй, что данные правильно сохраняются и извлекаются
   - Тестируй фильтрацию, сортировку, пагинацию

7. **Именование тестов:**
   - Формат: `MethodName_Scenario_ExpectedBehavior`
   - Пример: `GetUserById_UserExists_ReturnsUserDto`
   - Пример: `CreateUser_InvalidEmail_ThrowsValidationException`

8. **Организация тестов:**
   - Создай отдельный тестовый класс для каждого тестируемого класса
   - Используй `[Fact]` для простых тестов
   - Используй `[Theory]` и `[InlineData]` для параметризованных тестов
   - Группируй связанные тесты с помощью region или вложенных классов

9. **Покрытие кода:**
   - Цель: минимум 80% покрытия для нового кода
   - Покрой все ветки кода (if/else, switch, try/catch)
   - Не забывай про тесты для приватных методов (через публичные методы, которые их вызывают)

10. **Best practices:**
    - Один тест = одна проверка (не тестируй несколько вещей в одном тесте)
    - Тесты должны быть независимыми друг от друга
    - Избегай дублирования кода в тестах (используй helper методы или fixture)
    - Тесты должны быть читаемыми и самодокументирующимися
    - Не тестируй фреймворк или библиотеки (только свой код)

**Формат ответа:**
1. **Анализ коммита:** Список всех новых/измененных методов, которые нужно покрыть тестами
2. **План тестирования:** Какие типы тестов будут написаны для каждого метода
3. **Код тестов:** Полный код тестовых классов с импортами
4. **Инструкция по запуску:** Команды для запуска тестов и генерации отчета о покрытии
5. **Оценка покрытия:** Примерный процент покрытия нового кода

**Критерии готовности:**
- Все новые/измененные методы покрыты тестами
- Тесты используют xUnit + Moq + FluentAssertions
- Каждый тест следует AAA паттерну
- Тесты изолированы и не зависят друг от друга
- Покрыты happy path, граничные случаи и обработка ошибок
- Именование тестов соответствует конвенции
- Тесты компилируются и проходят
- В конце ответа добавлена проверочная фраза: "Тесты для коммита Lineman написаны с учетом контекста проекта."
